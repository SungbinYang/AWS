  * 물리적인 서버는 존재하지만 논리적인 서버는 존재하지 않는다.
  * 서버를 관리하는 주체가 우리가 아니다.
- 클라우드 컴퓨팅의 역사
  * 홈페이지, 프로모션, 쇼핑몰
  * 과거에는 사내에 직접 물리서버를 두고 관리하고 구축을 하였다.
  * 사내에서 직접 관리하면 회사입장에서는 큰 부담이었다. 왜냐하면 사용자의 증가에따라 트래픽이 증가하고 우리는 그것에 대한 대처를 해야한다.
  
  ![스크린샷 2021-10-11 오후 3 17 35](https://user-images.githubusercontent.com/18282470/136741782-3558262d-6b53-4a3b-a59d-c9176508bff8.png)
  
  * 이에 따라 서버를 전문적으로 관리하게 되는 Data center가 생겨남
  
  ![스크린샷 2021-10-11 오후 3 19 03](https://user-images.githubusercontent.com/18282470/136741932-d6131637-1aa6-4300-9eac-d1dafd3ce5a3.png)
  
  * 데이터 센터 대응방식은 트래픽이 많은 서버스들을 물리적으로 스케일 업하거나 스케일 아웃해서 확장이 필요한 서비스들을 새로운 서버에 설치하고 서비스 요청을 물리적 서버로 분산시킨다.
  * 시간이 지나서 서버가상화 기술이 도입됨. 이런 서버 가상화를 임대해주는게 클라우드 서비스 회사의 서비스이다.
  * 위와 같은 형태를 IaaS (Infrastructure as a Service)라고 한다. ex). AWS의 EC2, RDS
  
  ![스크린샷 2021-10-11 오후 3 23 54](https://user-images.githubusercontent.com/18282470/136742339-97a780cd-d145-42eb-85c7-3139052f0c28.png)
  
  ![스크린샷 2021-10-11 오후 3 25 21](https://user-images.githubusercontent.com/18282470/136742446-adf13f53-af4c-419a-8447-34a52a352fd1.png)
  
  * 위의 그림과 같은 형태를 PaaS (Platform as a Service)라고 한다. ex). AWS의 Elastic Beanstalk, Google firebase
  * IaaS (효율적안 인프라 제공) -> PaaS (다양하고 편리한 플랫폼 제공)
  * IaaS -> PaaS | Monolithic -> MSA
  * 사용자 어플리케이션은 후에 비즈니스 단위에서 함수단위로 쪼개지게 되는게 그것을 서버리스라고 부른다. :: FaaS (Function as a Service) ex). AWS의 Lambda
  * FaaS의 장점은 유연한 확장이 가능하다.
  * PaaS (다양하고 편리한 플랫폼 제공) -> FaaS (확장성과 신속하고 효율적인 대응)
- 서버리스의 개념
  * 비즈니스 로직에 집중하고, 서버운영으로부터 자유로워지는 클라우드 컴퓨팅 환경
